# RYZECHØ プロジェクト再設計メモ

最終更新: 2026-05-05

この文書は、MonoGame プロトタイプを今後も壊さず拡張するための再設計方針を整理します。正本仕様は引き続き `docs/game-spec.md`、実装差分は `docs/prototype-alignment.md` を優先します。

## 目的

- `GameModel` partial に集中しているルール値、初期ロスター、シミュレーション、描画責務を段階的に分離する
- 既存のプレイ可能状態を維持したまま、エージェント追加、武器調整、Unity 側移植の材料を取り出しやすくする
- 仕様値や固定データをコード全体に散らさず、検証と差し替えの入口を明確にする

## 現在の境界

- `Domain`: ゲーム内の列挙型、アクター / 武器 / 構築物モデル、固定ロスター、ルール値
- `World`: グリッド、座標変換、マップ構造、武器 / 構築物カタログ
- `Simulation`: ラウンド進行、AI、ボム処理、ボス投資、共有視界
- `Rendering`: HUD、ワールド、アクター、ショップ、描画ヘルパー
- `Input`: MonoGame 入力からゲーム入力スナップショットへの変換
- `Integrity`: ローカル保存と実行時状態の整合性検証

## 今回の基盤変更

- 画面寸法、グリッド、試合ルール、経済、視野、ボム秒数を `GameLayout` / `GameRules` に集約
- 固定味方ロスターを `RosterCatalog` と `ActorBlueprint` に集約
- `GameModel` のコンストラクタから固定アクター生成データを外し、ブループリントから生成する形に変更
- ボス選択 UI と整合性検証で、固定名の直書きを `RosterCatalog` 参照へ置き換え

## 次の分離候補

1. `WeaponStats` と構築物データを JSON か専用カタログへ寄せ、仕様値の差し替えを容易にする
2. `GameModel` の状態を `MatchState`、`EconomyState`、`ObjectiveState` に分割し、整合性検証の対象を明確にする
3. `Rendering` から `GameModel` の内部状態直接参照を減らし、読み取り専用の HUD / world snapshot を渡す
4. AI の経路選択と戦闘判断を `Simulation` 内の専用サービスへ分ける
5. 保存データとアンチチート系をプロトタイプ専用層として明示し、将来のサーバー権威実装と混同しないようにする
